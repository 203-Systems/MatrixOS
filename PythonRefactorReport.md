## 3. Design Decisions Required

Before implementing changes, these decisions must be made:

### 3a. Keep legacy integer keyID or switch to InputId?

The current Python API exposes keypad keys as a single `uint16_t` integer. The
new C++ API uses `InputId` (clusterId + memberId pair).

- **Option A**: Keep integer keyID in Python, translate internally via
  `BridgeKeyId`/`InputIdToLegacyKeyId`. Minimal Python-facing change, but
  preserves the legacy concept.
- **Option B**: Expose `InputId` to Python (two integers: cluster + member).
  Breaking change for Python scripts, but aligns with the new architecture.

### 3b. Rename Python modules?

- `KeyPad` → `Input`
- `KeyEvent` → `InputEvent`
- `KeyInfo` → `InputInfo` or `KeypadInfo`
- `KeyState` → `InputState` or `KeypadState`

### 3c. Expose inputClass to Python?

The C++ layer now requires explicit `inputClass` checks. Python scripts should
either:
- Receive pre-filtered keypad-only events (simple, backward-compatible)
- Receive all events with an `inputClass` field (more powerful, breaking)

### 3d. Scope of Python API redesign

The PostInputSystemAction.md notes: "the current python API wrapper is not
ideal, stable, safe, or designed well so we need a redesign." If a full redesign
is planned, the input system migration could be done as part of that effort
rather than a minimal patch.

## 4. Recommended Migration Plan

### Phase 1: Minimal compilation fixes (unblock Python build)

Fix the 7 compilation blockers without changing the Python API surface:

1. `MatrixOS_KeyPad.cpp`:
   - `GetKeypadState(Point)` → `GetInputAt()` + `GetState()` + `.keypad`
   - `TryGetPoint()` → `GetPosition()`
   - Keep `BridgeKeyId`/`InputIdToLegacyKeyId` temporarily

2. `MatrixOS_UI.cpp`:
   - `SetKeyEventHandler()` → `SetInputEventHandler()`
   - `TryGetPoint()` → `GetPosition()`
   - `IsFunctionKey()` → `InputId::FunctionKey()` comparison
   - Remove `FUNCTION_KEY` constant usage

### Phase 2: Remove legacy keyID translation layer

Once Phase 1 compiles, remove `BridgeKeyId` and `InputIdToLegacyKeyId`:

- Switch `PythonKeyEvent.id` from `uint16_t` to `InputId`
- Update `MatrixOS_KeyEvent.cpp` to expose `ClusterId()` and `MemberId()`
- Update `.pyi` stubs and `.py` modules
- Re-run pika compiler

### Phase 3: Full Python API redesign (optional, larger scope)

If a full redesign is approved:

- Rename modules (`KeyPad` → `Input`, etc.)
- Expose `inputClass` and cluster concepts
- Add `GetClusters()`, `GetPrimaryGridCluster()` to Python
- Design a proper event model instead of the polling `Get()` loop
- Add type safety and error handling
- Remove the `PythonKeyEvent` bridge struct entirely
